TypeScript的Classes 与 Interfaces

2017-09-17
typescript

文章目录

    • 例子
    • 接口
    • 类
    • 结论

本文原文地址，翻译如有问题请指正

我们现在已经可以在JavaScript中使用类了，在TypeScript中类早已经支持了。但是在TypeScript中我们有一个interface的概念，那么当我们添加type annotations的时候，经常会有疑问：

这个type annotations应该使用interface还是class

本文将会对比1TypeScript中的classes和interfaces的区别，来解答这个问题。

例子

fetch('https://jameshenry.blog/foo/bar')
    .then((response) => {
        console.log(response.status) // Some HTTP status code, such as 200
    })

上面的代码从远端的服务器获取数据，然后使用这些数据在前端的代码中。

如果TypeScript来编译这个代码，那么它可能会将参数response类型推断为any。

那么为了提升我们程序的类型安全，我们显式的给出response的类型

// `Response` will be defined here...

fetch('https://jameshenry.blog/foo/bar')
    .then((response: Response) => {
        console.log(response.status)
    })

现在到了我们前文提出的问题，这个Response类型应该定义成一个interface还是一个class?

接口

当TypeScript检查程序的各个部分的类型的时候，它所使用的方法是鸭子类型

如果它看上去是一个鸭子，并且叫起来也像鸭子，它就是一个鸭子。

换句话说，我们是否可以通过一些特定的特征/结构/形状来确定一个特定的类型。从现在起我们称作它为shape。

在TypeScript中，一个interface是给定了名字的一些特定的shape的组合，那么在程序中引用它。

看下一个鸭子的接口：

// A duck must have...
interface Duck {
    // ...a `hasWings` property with the value `true` (boolean literal type)
    hasWings: true
    // ...a `noOfFeet` property with the value `2` (number literal type)
    noOfFeet: 2
    // ...a `quack` method which does not return anything
    quack(): void
}

现在我们的代码中，如果我们想要确定某些类型是不是duck（其实就是，它必须实现我们的Duck接口），我们定义个Duck类型的变量

// This would pass type-checking!
const duck: Duck = {
    hasWings: true,
    noOfFeet: 2,
    quack() {
        console.log('Quack!')
    },
}

// This would not pass type-checking as it does not
// correctly implement the Duck interface.
const notADuck: Duck = {}
// The TypeScript compiler would tell us
// "Type '{}' is not assignable to type 'Duck'.
// Property 'hasWings' is missing in type '{}'."

下面我们讨论下，在编译阶段interfaces是如何被编译的，首先我们必须把握一个原则：

interface只会在编译阶段被typescript使用，然后它就会被移除。Interfaces不会出现在最终的JavaScript代码中。

下面是定义了Response接口的完整代码

interface Response {
    status: number // Some HTTP status code, such as 200
}

fetch('https://jameshenry.blog/foo/bar')
    .then((response: Response) => {
        console.log(response.status)
    })

如果我们编译这个代码，会生成下面的JavaScript的代码：

fetch('https://jameshenry.blog/foo/bar')
    .then(function (response) {
    console.log(response.status);
});

response已经被删除掉了，它没有在最终的代码中。

类

现在我们重新定义Response为class：

class Response {
    status: number // Some HTTP status code, such as 200
}

fetch('https://jameshenry.blog/foo/bar')
    .then((response: Response) => {
        console.log(response.status)
    })

如你所见，我们只是简单的把interface改成了class。如果我们编译这段代码，我们的类型是安全的也没有编译错误。

实际中它们两者的区别可以看作就是在JavaScript的输出。

不像interface，一个class在JavaScript也是一个construct，它不仅仅是一个类型的命名。

它们两者最大的区别是class提供了一个实现，而不是仅仅是一个shape

下面是将interface改为class后编译输出的代码:

var Response = (function () {
    function Response() {
    }
    return Response;
}());
fetch('https://jameshenry.blog/foo/bar')
    .then(function (response) {
    console.log(response.status);
});

非常的不一样！可以看到，我们的class被以ES5相兼容的函数形式来表现，在我们最终的JavaScript程序中这是不必要的部分。如果我们有一个很大的程序，如果重复使用类作为类型声明，那么我们的最终的包会大很多。

结论

如果我们需要从远端的服务器来获取js，那么使用一个interface是个好主意。

因为在最终的JavaScript的代码中，interface会被完全的移除

如果我们需要来实现接口，那么也可以非常容易的完成一个完整的类。
